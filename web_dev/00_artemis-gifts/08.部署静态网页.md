### 📝 技术笔记 (一)：React 项目的构建与云端部署

**核心逻辑：**
写代码只是第一步，浏览器看不懂 `.jsx` 文件。我们需要通过“构建 (Build)”把代码翻译成浏览器能懂的 HTML/CSS/JS，然后把这些文件“托管 (Host)”到服务器上。

#### 1. 关键概念：源码 vs 产物

* **Source Code (源码):** 你在 VS Code 里写的代码 (`src/App.jsx`)。浏览器无法直接运行。
* **Build Artifacts (构建产物):** 运行 `npm run build` 后生成的 **`dist`** 文件夹。
* 里面通常只有一个 `index.html` 和一堆加密混淆过的 `.js` 文件。
* **重点：** 我们最终发布到网上的，永远是这个 `dist` 文件夹，而不是你的源代码。



#### 2. Cloudflare Pages 部署避坑指南

Cloudflare 是目前国内访问最友好的免费静态托管平台之一，但配置上有两个巨大的“坑”：

* **💣 坑位一：Output Directory (输出目录)**
* **现象：** 部署成功，但打开是 "Hello World"、空白页或 README 文档。
* **原因：** Cloudflare 默认不知道你的成品在哪里，它可能直接把你的源码根目录发布了。
* **正解：** 必须在设置里将 `Build output directory` 明确填为 **`dist`** (Vite 项目的标准输出目录)。


* **💣 坑位二：多余的 Deploy Command**
* **现象：** 部署直接报错 "Error: Missing entry-point"。
* **原因：** Pages 是自动化托管，不需要手动写上传命令。如果你填了 `wrangler deploy`，系统会以为你要运行后端 Worker 脚本。
* **正解：** `Deploy command` 这一栏必须 **留空 (Optional)**。



#### 3. 两种部署流派

* **流派 A：Git 自动化 (CI/CD)**
* **原理：** 本地 Push 代码 -> GitHub 收到更新 -> 通知 Cloudflare -> Cloudflare 自动拉取代码并运行 `npm run build` -> 发布。
* **优点：** 一劳永逸，改代码只需 Push。
* **注意：** 必须确保 Cloudflare 绑定的**分支 (Branch)** 和你本地 Push 的分支一致（比如都是 `main` 或 `master`）。


* **流派 B：手动拖拽 (Direct Upload) —— 你的“救命稻草”**
* **原理：** 绕过 GitHub，直接把本地生成的 `dist` 文件夹拖给 Cloudflare。
* **优点：** **100% 成功率**，不依赖 Git，不受分支混乱影响，配置错误率为 0。适合紧急发布或 Git 搞不定的时候。



#### 4. SPA 应用的“刷新 404”魔法

* **问题：** 单页应用 (React/Vue) 只有一张 `index.html`。当用户访问 `/secret` 路由时，如果直接刷新，服务器会找不到 `secret.html` 而报 404。
* **解法：** 在 `public` 文件夹下新建 **`_redirects`** 文件。
* **代码：** `/* /index.html 200`
* **含义：** “不管用户请求什么路径，都把 `index.html` 给它，让 React 自己的路由去处理。”
